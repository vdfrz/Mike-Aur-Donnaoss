### resolve_precedents fans out external Kanoon network calls in a loop with no per-call or total timeout

Each call also does at most MAX_PAGES=3 sequential search requests plus fragment fetches run concurrently via `futures_util::future::join_all` (line 191), so per-precedent latency is bounded; the only residual is that `required_precedents` length is uncapped and the loop at cases.rs:890 is sequential, giving finite (not infinite) linear latency — which is not the unbounded/indefinite-hang problem the finding describes.

### Literal find/replace runs over the entire document.xml, including markup, not just visible text
- **File:** `src/pdf/docx_writer.rs:656-667`  ·  **human-logic**  ·  sev HIGH → **NOT_A_BUG**
- **Problem:** patch_document_xml searches `working` (the whole word/document.xml string, tags and attributes included) for the escaped needle and replaces every occurrence. - **Fix:** Restrict literal replacement to visible `<w:t>…</w:t>` text spans (the same run-scoped approach tolerant_replace_in_runs uses), or require a minimum needle length / word-boundary match so single tokens can't collide with markup; never replace across raw tag bytes.

The live edit_document tool routes through apply_tracked_edits → patch_document_xml_tracked → tracked_replace_in_run, which extracts run_visible_text(run_xml) first and matches the needle only within that visible <w:t> text (docx_writer.rs:928-930) — exactly the mitigation the finding proposes. So the LLM-supplied `find` never reaches the buggy code path; the vulnerability is unreachable dead code, not an exploitable bug.

### zip(chunks, vectors) silently drops chunks if embed returns fewer vectors than inputs, but reports full count

### embed_passages never verifies it produced one vector per input passage
- **File:** `src/embeddings/service.rs:284-330`  ·  **human-logic**  ·  sev HIGH → **NOT_A_BUG**
- **Problem:** The batch loop extends all_vectors with each batch's output but never checks all_vectors.len() == n (the prefixed input length) before returning at line 330.
