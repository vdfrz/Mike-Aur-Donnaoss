## The one fact that shapes everything

**Pandoc reads tracked changes; it does NOT write them.** `--track-changes=accept|reject|all` only
affects the docx *reader*; pandoc's docx *writer* emits a clean document with no `w:ins`/`w:del`.
So "run pandoc to produce the redline" is impossible. Pandoc's real job here is **fidelity of the text
the model reads** (and, with `--track-changes=all`, letting the model *see* an already-marked-up doc's
insertions+deletions — which the current Rust extractor drops, see `docs/DOCX.md:81`). The redline
itself (`w:ins`/`w:del`) is generated by **our Rust writer**, upgraded from verbatim find/replace to a
**word-level diff** so edits actually land and insertions are possible.

```
  READ  (pandoc augments, redline path only)        WRITE (Rust, diff-driven — replaces find/replace)
  .docx ─► pandoc docx→markdown --track-changes=all   model's revised clause text
          (structured: clauses, lists, tables)   ─►   word-diff(original, revised)
          fallback: extract_docx_text (pure Rust)      └─► <w:del>/<w:ins> per hunk (incl. pure insert)
                                                            accept/reject + frontend + bot UNCHANGED
```

## Decisions taken (auto mode — both reversible; flip in §0 if you disagree)
1. **Augment, not `pandiff`-owns-it.** Tracked-change generation stays in Rust (Loop P-W). Rejected the
   `pandiff` (pandoc+Node CriticMarkup→OOXML) route: adds a Node runtime + external pipeline, prose-grade
   not surgical-clause-grade, and conflicts hardest with `docs/DOCX.md`. If you later want pandoc to own
   the whole redline, that's a separate Loop P-X — not in this plan.
2. **Capability layer + graceful fallback first; Tauri bundling last (Loop P-B).** No hard dependency:
   `pandoc_available()` gates every call; absent pandoc → the existing pure-Rust extractor. The redline
   degrades in *input fidelity*, never breaks.

## Reconciling `docs/DOCX.md` ("no Pandoc")
DOCX.md's three objections — 300 MB binary, ~1 s spawn per extraction, mammoth dropping redlines — all
target the **hot extraction path**: the folder scanner, the upload cache, and `read_document`
(high-frequency, embedding-bound). The **redline path is the opposite**: user-initiated, one contract at
a time, latency-tolerant. So: **keep pure-Rust `extract_docx_text` as the default for indexing/RAG; add
pandoc only on the redline path, behind a capability check.** Loop P-B rewrites the DOCX.md claim to
*"pure Rust for the extraction hot path; pandoc is an optional, bundled converter scoped to the
redline-quality path."* This is an **augment**, and the file's own objections still hold for the path
they were written about.
